Boost Agents Setup Guide

Everything you need to create, configure, and deliver an AI agent to a client.

Overview

Each agent is a GitHub repo created from the MoveoTech/boost-agents template. Pushing to main automatically provisions a GCP project and deploys a single Cloud Run service that serves both the API and the web UI from one URL.

Users log in with Google. Admins (set via ADMIN_EMAILS secret) see a full configuration sidebar. Regular users see the chat and their personal Google connections. Sessions persist across tabs and browser restarts for 7 days.

ℹ
The oauth-service is deployed once at boost-agents-496211 and shared across all agents. You never need to redeploy it per client.

1. Create a repo

1
Go to the template repo Open github.com/MoveoTech/boost-agents and click Use this template → Create a new repository.
2
Name it Use the pattern boost-<client>-<use-case>, e.g. boost-acme-hr-agent. The GCP project is named after the repo automatically.
3
Set owner to MoveoTech Make sure the owner is MoveoTech (not your personal account) so org secrets are inherited automatically.

2. Add secrets

Go to your new repo → Settings → Secrets and variables → Actions → New repository secret.

SecretRequiredWhere to get it
GEMINI_API_KEY Required aistudio.google.com → Get API key
API_KEY Required Any strong random string — used as the admin API key for programmatic access via x-api-key header
ADMIN_EMAILS Required Comma-separated emails of admins, e.g. you@moveoboost.com,client@acme.com. These users see the full config sidebar.
COOKIE_SECRET Required Any long random string (32+ chars). Used to sign session JWTs. Without it the server falls back to an insecure default.
GH_PAT Org secret Inherited automatically — no action needed
OAUTH_SERVICE_URL Org secret Inherited automatically — no action needed
OAUTH_SERVICE_KEY Org secret Inherited automatically — no action needed
AUTOMATION_SECRET Org secret Inherited automatically — no action needed
ANTHROPIC_API_KEY Optional console.anthropic.com — only needed if using Claude models
OPENAI_API_KEY Optional platform.openai.com — only needed if using GPT/o1 models

3. Deploy

Push anything to main to trigger the deploy. The first run takes ~5–8 minutes and will:

  1. Create a GCP project named boost-<repo-name>-v7
  2. Enable all required APIs (Cloud Run, Artifact Registry, Cloud Scheduler, etc.)
  3. Build the web app and bundle it into the server image
  4. Deploy a single Cloud Run service that serves both the UI and API

The agent URL is printed at the end of the Action run under Print URL. This single URL serves everything — share it directly with users.

⚠
If the Action fails on first run, just re-run it. The provisioning step has a timing dependency on API activation — a second run almost always succeeds.

4. Configure the agent

Open the agent URL and sign in with your Google account. As an admin you'll see the configuration sidebar on the right.

Model

Each user picks their own model — changes save personally to their browser and don't affect other users. Available models depend on which API keys are set as repo secrets. Admins can click Publish Changes to set a new global default for users who haven't picked their own.

Instructions

The agent's system prompt. Like the model, changes are personal by default. Admins can publish to set the global baseline. Users can personalise on top of whatever the admin has set.

Tools

Admins control which tools are globally available (toggles). Google tools (Gmail, Calendar) are automatically active as soon as a user connects their account — no extra toggle needed.

Connections

Every user connects their own Google account via the Connect link. Once connected, Gmail and Calendar tools work immediately and stay connected across sessions.

Skills

Knowledge blocks appended to the system prompt. The agent uses them automatically when relevant — think SOPs, product documentation, FAQs. Enable/disable per skill without redeploying.

Automations

Scheduled tasks powered by Cloud Scheduler. Each automation has a name, cron schedule, and a prompt. At the scheduled time the agent runs the prompt for every connected user. Each automation shows a tag with who created it.

Publish Changes

Commits the global config (model default, instructions, tool toggles, skills) to GitHub, making it the permanent baseline for all new users and future deploys. Changes take effect immediately in the current session regardless of whether you publish.

Delivering to a client

1
Create and deploy the repo Follow steps 1–3 above. Add the client's admin email to ADMIN_EMAILS alongside your own.
2
Add client users as OAuth test users Until Google verification is complete, add each user's Gmail manually. See Google OAuth setup below.
3
Configure the agent Log in, write the system prompt, enable tools, add skills. Click Publish Changes to lock in the config permanently.
4
Send the URL to the client One URL — serves the full web UI. Users sign in with Google, connect their Gmail/Calendar, and start chatting.

Google OAuth setup

The OAuth app lives in GCP project boost-agents-496211 — one app shared by all agents.

Adding test users (before verification)

In Testing mode, only explicitly added emails can connect Gmail/Calendar:

  1. Go to console.cloud.google.com
  2. Select project boost-agents-496211
  3. Navigate to APIs & Services → OAuth consent screen
  4. Scroll to Test users → click + Add users
  5. Enter the client's Gmail address and save
⚠
Maximum 100 test users. Fine for pilots — once you need more, proceed with Google verification.

Google verification

Allows any Google account to connect Gmail and Calendar without being added as a test user. One-time process for the central OAuth app.

Prerequisites

  • Public privacy policy URL — e.g. moveoboost.com/privacy. Must explain what data is accessed and why.
  • Public homepage URL — your product page
  • App logo — 120×120px PNG minimum
  • Domain verified in Google Search Console — search.google.com/search-console, add moveoboost.com, verify via DNS TXT record

Steps in GCP Console

1
Open the consent screen Go to boost-agents-496211 → APIs & Services → OAuth consent screen. Confirm it's set to External.
2
Fill in app information
  • App name: Boost Agents
  • User support email: boazt@moveoboost.com
  • App logo: 120×120px PNG
  • Homepage: https://moveoboost.com
  • Privacy policy: https://moveoboost.com/privacy
  • Authorized domain: moveoboost.com
3
Add scopes — add exactly these, no more:
  • …/auth/userinfo.email Know who the user is (login)
  • …/auth/gmail.send Send emails on the user's behalf
  • …/auth/calendar.readonly Read calendar events
  • …/auth/calendar.events Create and update calendar events

Note: gmail.readonly and gmail.metadata are restricted scopes requiring a costly security audit — they are intentionally excluded.

4
Submit for verification Click Submit for verification. Google will ask you to justify each scope.

Scope justifications (copy-paste)

ScopeJustification to submit to Google
gmail.send The agent sends emails on behalf of the user when explicitly instructed in the chat (e.g. "send a summary to my team"). The user triggers each send action.
calendar.readonly The agent reads calendar events to answer scheduling questions (e.g. "what do I have tomorrow?"). No data is stored or shared with third parties.
calendar.events The agent creates calendar events when the user explicitly requests it (e.g. "schedule a meeting with John next Tuesday at 3pm"). Events are only created on direct user instruction.

Slack bot setup

Each agent can have its own Slack bot. Users mention the bot in a channel and it responds in-thread with full access to Gmail, Calendar, and all other configured tools — exactly the same capabilities as the web chat.

ℹ
Each agent repo gets its own Slack app. The bot's Slack email is matched against the user's connected Google account to load their Gmail and Calendar access automatically.

Step 1 — Create a Slack app

  1. Go to api.slack.com/apps → Create New App → From scratch
  2. Give it a name (e.g. HR Agent) and pick your workspace

Step 2 — Add bot scopes

Under OAuth & Permissions → Bot Token Scopes, add:

ScopeWhy
chat:writePost messages to channels
channels:readList public channels
groups:readList private channels (optional)
users:readLook up user info by ID
users:read.emailFetch the user's email to load their Gmail/Calendar connection

Step 3 — Install the app

Under OAuth & Permissions → click Install to Workspace. Copy the Bot User OAuth Token (xoxb-...).

Step 4 — Enable Event Subscriptions

  1. Under Event Subscriptions, toggle Enable Events on
  2. Set the Request URL to https://your-agent-url/slack/events — Slack will verify it automatically once your agent is deployed
  3. Under Subscribe to bot events, add app_mention
  4. Save changes and reinstall the app if prompted

Step 5 — Add repo secrets

In the agent repo → Settings → Secrets → Actions, add:

SecretWhere to get it
SLACK_BOT_TOKENOAuth & Permissions → Bot User OAuth Token (xoxb-...)
SLACK_SIGNING_SECRETBasic Information → App Credentials → Signing Secret

Step 6 — Enable Slack in the agent sidebar

Log in to the agent, go to Tools, toggle Slack on, and click Publish Changes.

Step 7 — Invite the bot to channels

In Slack, type /invite @YourBotName in each channel you want it to respond in.

Using the bot

Mention the bot anywhere it's been invited:

@HRAgent send an email to john@acme.com summarising today's agenda
@HRAgent what meetings do I have this afternoon?
@HRAgent schedule a 1:1 with sarah@acme.com next Tuesday at 10am

The bot replies in-thread. It uses the Slack user's email to find their connected Gmail and Calendar — so each user gets their own personalised access.

⚠
Gmail and Calendar only work for users who have connected their Google account via the agent's web UI. If a user hasn't connected, the bot will tell them to do so.

One bot per agent

Each agent repo has its own Slack app. If you have three agents (HR, Sales, Finance), you create three Slack apps — each pointing to its own agent URL and installed in the workspace. They appear as separate bots in Slack.

API access

The agent exposes a REST API. Authenticate with the x-api-key header (value = API_KEY secret).

Send a message

curl -X POST https://your-agent-url/api/chat \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{"message": "Send a welcome email to john@acme.com", "history": [], "gmailToken": "GMAIL_JWT"}'

Get a Gmail/Calendar token for a connected user

curl "https://your-agent-url/api/google-token?email=user@example.com&service=gmail" \
  -H "x-api-key: YOUR_API_KEY"

Response format

{
  "reply": "Email sent to john@acme.com.",
  "toolUses": [
    { "name": "gmail_send", "input": "...", "output": "..." }
  ]
}
