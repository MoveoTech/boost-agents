Boost Agents Setup Guide

Everything you need to create, configure, and operate AI agents.

Overview

Boost Agents is a platform for deploying AI agents powered by Gemini, Claude, or OpenAI. Each agent is its own GitHub repo — push to main and the agent auto-deploys to a dedicated GCP project with a single URL serving both the server and the web UI.

Agents connect to Gmail, Google Calendar, Google Tasks, Monday.com, Slack, and WhatsApp. They support file attachments, slash command shortcuts, scheduled automations, persistent memory, and chat history.

Creating an agent

1
Create a repo from the template In GitHub, go to MoveoTech/boost-agents → Use this template → Create a new repository. Name it anything — the agent URL will use this name.
2
Add the GEMINI_API_KEY repo secret In the new repo: Settings → Secrets and variables → Actions → New repository secret. Add GEMINI_API_KEY from aistudio.google.com. All other secrets (OAUTH_SERVICE_URL, OAUTH_SERVICE_KEY, GH_PAT) inherit automatically from the org.
3
Push to main The deploy pipeline runs automatically. First deploy takes ~8 minutes (GCP provisioning). Subsequent deploys take ~2 minutes.
4
Find your URL GitHub → Actions tab → latest workflow run → the URL appears at the top of the summary under "✦ Agent deployed". Also visible in GCP Console → Cloud Run → boost-<name>-v7.
⚠
If the first deploy fails, go to Actions → click the failed run → Re-run all jobs. A timing issue during GCP provisioning occasionally affects the first run only.

Configuring the agent

Open the agent URL, sign in with Google. Admins see the configuration sidebar on the right.

Model & Instructions

Each user picks their own model and system prompt — changes are personal. Click Publish Changes to commit the global default for all users and future deploys.

Tools

Toggle tools on/off globally. Google tools (Gmail, Calendar, Tasks) activate per-user once they connect their account — no extra admin toggle needed.

Skills

Knowledge blocks always present in the agent's context — SOPs, product docs, FAQs. Enable/disable without redeploying.

Automations

Scheduled prompts that run on a cron schedule. The agent runs using each connected user's own Gmail/Calendar access.

Starter prompts

Buttons shown on the empty chat screen. Each becomes a clickable prompt for new users.

Connecting services

Each user connects their own accounts from the Settings sidebar. Connections are per-user and persist across sessions.

Gmail & Google Calendar & Tasks

Click Connect Gmail / Connect Calendar / Connect Tasks in the sidebar → Google OAuth flow. Users must be added to the OAuth test users list by Boaz — see Adding OAuth clients.

Monday.com

Click Connect Monday → Monday OAuth flow. The app requests these scopes:

  • boards:read boards:write — read and write boards, items, columns, groups
  • updates:read updates:write — read and post comments/updates
  • users:read — look up user IDs for person column assignments
  • account:read workspaces:read — account and workspace info
  • me:read — current user's profile
⚠
If Monday.com scopes change in the code, existing users must disconnect and reconnect to get the updated token with the new scopes.

WhatsApp connector

Each agent can be connected to a personal WhatsApp number. The bot runs as a linked device on that number using the Baileys library.

Setup

1
Open Settings → Connect WhatsApp A QR code appears. Scan it from WhatsApp on your phone: Settings → Linked devices → Link a device.
2
Configure the trigger Choose how the bot decides to reply: keyword (e.g. messages containing "boost"), @mention, or always. Enable ownerOnly to have the bot only respond to messages sent from the connected account itself (useful for personal assistants).
3
Choose a model WhatsApp defaults to Gemini 2.5 Flash. You can override per-agent in the settings.

Limitations

⚠
One WhatsApp connection per agent. Only one phone number can be linked at a time. If a second user tries to connect, they'll see "already connected to {email}".
  • Personal number only — uses the Baileys linked device protocol, not the official WhatsApp Business API. The connected number must be a personal WhatsApp account.
  • Session is a linked device — if the main phone logs out all linked devices (WhatsApp Settings → Linked Devices → Log out all), you'll need to re-scan the QR code.
  • Signal session instability — groups with many participants generate a lot of Signal key exchanges. The connector auto-recovers from most errors, but occasional reconnects are normal.
  • Attachments in quoted messages — when replying to someone else's document, the bot can download and analyze it if the CDN URL hasn't expired. If download fails, resend the file directly.

Slack bot setup

Each agent can have its own Slack bot. Mention the bot to trigger the agent with full Gmail/Calendar access.

Step 1 — Create a Slack app

  1. Go to api.slack.com/apps → Create New App → From scratch
  2. Name it and select your workspace → Create App

Step 2 — Add bot scopes

Under OAuth & Permissions → Bot Token Scopes, add: chat:write, channels:read, groups:read, users:read, users:read.email

Step 3 — Install and copy token

Under OAuth & Permissions → Install to Workspace. Copy the Bot User OAuth Token (xoxb-...) — this is SLACK_BOT_TOKEN.

Step 4 — Enable Event Subscriptions

  1. Toggle Enable Events on
  2. Set Request URL to https://your-agent-url/slack/events
  3. Subscribe to bot event: app_mention
  4. Save and reinstall if prompted

Step 5 — Get the Signing Secret

Basic Information → App Credentials → Signing Secret → Show — this is SLACK_SIGNING_SECRET.

Step 6 — Add repo secrets

Add SLACK_BOT_TOKEN and SLACK_SIGNING_SECRET as GitHub repo secrets and redeploy.

Step 7 — Enable Slack in the sidebar

Log in → Tools → toggle Slack on → Publish Changes.

Step 8 — Invite bot to channels

In Slack: channel name → Integrations → Add an App → find your bot.

Security model

All agents share a central oauth-service running in boost-agents-496211. Data is separated by agentId (= the GCP project name) in Firestore. Access is enforced at the API level — each agent gets a unique scoped key.

How per-agent keys work

  • The master key (OAUTH_SERVICE_KEY org secret) is used only by CI/CD and the oauth-service itself — never given to agents or clients.
  • On first deploy, CI/CD calls POST /api/agent-keys with the master key to provision a unique key for the new agent, stores it in that agent's GCP Secret Manager, and injects it as OAUTH_SERVICE_KEY in Cloud Run.
  • A per-agent key can only access data for its own agentId. Attempting to read another agent's data returns 403.
  • A leaked per-agent key exposes only that one agent's data — not the entire platform.

What's still shared

  • Firestore database (data isolated by agentId, but same quota pool)
  • oauth-service Cloud Run instance (single point of failure)

For clients requiring full infrastructure isolation, see Client deployment options.

Creating agents at scale

With org-level secrets in place, creating a new agent is three steps:

1
Create repo from template MoveoTech/boost-agents → Use this template.
2
Add GEMINI_API_KEY Repo → Settings → Secrets → New repository secret → GEMINI_API_KEY. Add any other optional API keys (Anthropic, OpenAI, Slack) if needed.
3
Push to main CI/CD provisions the GCP project, per-agent OAuth key, builds the image, and deploys. URL appears in Actions summary.
ℹ
Each agent's per-agent OAuth key is auto-generated and stored in the agent's own GCP Secret Manager (oauth-agent-key). You don't manage these keys manually.

Customising the agent

Edit apps/server/src/config.ts in the child repo to set the agent name, system prompt, enabled tools, model, and UI settings. Push to main to redeploy with the new config.

API access

Authenticate with the x-api-key header using the agent's API_KEY (find it in GCP Secret Manager → api-key).

Basic request

curl -X POST https://your-agent-url/api/chat \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{"message": "Hello!", "history": []}'

With Gmail / Calendar access

curl -X POST https://your-agent-url/api/chat \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{
    "message": "What meetings do I have today?",
    "history": [],
    "userEmail": "user@example.com"
  }'

Streaming

curl -X POST https://your-agent-url/api/chat \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{"message": "Summarize my emails", "history": [], "stream": true}'

Streaming returns text/event-stream with data: {type, content} lines. Types: token, tool_start, tool_complete, done, error.

Response format (non-streaming)

{
  "reply": "You have 3 meetings today…",
  "toolUses": [
    { "name": "calendar_list_events", "input": "...", "output": "..." }
  ]
}
