Boost Agents Setup Guide

Everything you need to create, configure, and deliver an AI agent to a client.

Overview

Each agent is a GitHub repo created from the MoveoTech/boost-agents template. Pushing to main automatically provisions a GCP project and deploys two Cloud Run services:

  • Server — Express API + AI agent (Gemini, Claude, or OpenAI)
  • Web — React chat UI with admin sidebar

Users log in with Google. Admins (set via ADMIN_EMAILS secret) see a configuration sidebar. Regular users see the chat and their personal Google connections.

ℹ
The oauth-service is deployed once at boost-agents-496211 and shared across all agents. You never need to redeploy it per client.

1. Create a repo

1
Go to the template repo Open github.com/MoveoTech/boost-agents and click Use this template → Create a new repository.
2
Name it Use the pattern boost-<client-name>-<use-case>, e.g. boost-acme-hr-agent. The GCP project will be named after the repo automatically.
3
Set owner to MoveoTech Make sure the owner is MoveoTech (not your personal account) so org secrets are inherited.

2. Add secrets

Go to your new repo → Settings → Secrets and variables → Actions → New repository secret.

SecretRequiredWhere to get it
GEMINI_API_KEY Required aistudio.google.com → Get API key
API_KEY Required Any strong random string — used as the admin API key for programmatic access
ADMIN_EMAILS Required Comma-separated emails of admins, e.g. you@moveoboost.com,client@acme.com. These users see the full config sidebar.
GH_PAT Org secret Inherited automatically — no action needed
OAUTH_SERVICE_URL Org secret Inherited automatically — no action needed
OAUTH_SERVICE_KEY Org secret Inherited automatically — no action needed
AUTOMATION_SECRET Org secret Inherited automatically — no action needed
ANTHROPIC_API_KEY Optional console.anthropic.com — only needed if using Claude models
OPENAI_API_KEY Optional platform.openai.com — only needed if using GPT/o1 models

3. Deploy

Push anything to main to trigger the deploy. The first run takes ~5–8 minutes and will:

  1. Create a new GCP project named boost-<repo-name>-v7
  2. Enable all required APIs (Cloud Run, Artifact Registry, Cloud Scheduler, etc.)
  3. Build and push Docker images
  4. Deploy the server and web app to Cloud Run

The URLs are printed at the end of the GitHub Action run under Print URLs. Share the Web URL with users.

⚠
If the Action fails on first run, just re-run it. The provisioning step has a timing dependency on API activation — a second run almost always succeeds.

4. Configure the agent

Open the web URL and sign in with your Google account. As an admin you'll see the configuration sidebar on the right.

Identity

Set the agent's name and upload an avatar. This is what users see at the top of the sidebar.

My Preferences

Personal settings that only affect your session — not other users. Choose a model and add personal instructions.

Model (global default)

The model all users default to. Users can override this in their own "My Preferences" section. Available models depend on which API keys are set as repo secrets.

Instructions (system prompt)

The agent's base behavior — its role, capabilities, tone, and constraints. This applies to all users unless they add personal instructions on top.

Tools

Enable or disable what the agent can do. Google tools (Gmail, Calendar) require each user to connect their account individually via the Connect link.

Skills

Reusable knowledge blocks appended to the system prompt. Use for SOPs, product documentation, or any reference content the agent should always have access to.

Automations

Scheduled tasks that run on a cron schedule. Each automation has a name, schedule, and prompt. The agent runs the prompt for every connected user at the scheduled time.

Publish Changes

Commits the global config to GitHub, which triggers a redeploy. Use this to make changes permanent across future deploys. Model and instructions take effect immediately without publishing — publishing only ensures they survive a redeploy.

Delivering to a client

1
Create and deploy the repo Follow steps 1–3 above. Add the client's admin email to ADMIN_EMAILS.
2
Add client users as OAuth test users Until Google verification is complete, external emails must be added manually. See the Google OAuth setup section below.
3
Configure the agent Log in, set the system prompt, enable the relevant tools, add skills. Use "Publish Changes" to lock in the config.
4
Send the web URL to the client Users sign in with Google. They see the chat and can connect their own Gmail/Calendar. Admins see the full sidebar.

Google OAuth setup

The OAuth app lives in GCP project boost-agents-496211. This is the central project — one app shared by all agents.

Adding test users (before verification)

During testing / before Google verification, you must add each user's Gmail address manually:

  1. Go to console.cloud.google.com
  2. Select project boost-agents-496211
  3. Navigate to APIs & Services → OAuth consent screen
  4. Scroll to Test users → click + Add users
  5. Enter the client's Gmail address and save
⚠
Maximum 100 test users. Sufficient for pilots — once you need more, proceed with Google verification.

Google verification

Required to allow any Google account (outside test users) to connect Gmail and Calendar. This is a one-time process for the central OAuth app.

Prerequisites — prepare these first

  • Public privacy policy URL — e.g. moveoboost.com/privacy. Must be live and explain what data is accessed and why.
  • Public homepage URL — your product page
  • App logo — 120×120px PNG minimum
  • Domain verified in Google Search Console — go to search.google.com/search-console, add moveoboost.com, verify via DNS TXT record

Steps in GCP Console

1
Open the consent screen Go to boost-agents-496211 → APIs & Services → OAuth consent screen. Confirm it's set to External.
2
Fill in app information
  • App name: Boost Agents
  • User support email: boazt@moveoboost.com
  • App logo: upload 120×120px PNG
  • Homepage: https://moveoboost.com
  • Privacy policy: https://moveoboost.com/privacy
  • Authorized domain: moveoboost.com
3
Add scopes Add exactly these (no more):
  • …/auth/userinfo.email Know who the user is
  • …/auth/gmail.send Send emails on the user's behalf
  • …/auth/gmail.readonly Read emails to answer questions
  • …/auth/calendar.readonly Read calendar events
  • …/auth/calendar.events Create and update calendar events
4
Submit for verification Click Submit for verification. Google will ask you to justify each scope — use the text below.

Scope justifications (copy-paste)

ScopeJustification to submit to Google
gmail.send The agent sends emails on behalf of the user when instructed in the chat (e.g. "send a summary to my team"). The user explicitly requests each send action.
gmail.readonly The agent reads emails to answer user questions and provide context (e.g. "summarize my unread emails from today"). No emails are stored or shared.
calendar.readonly The agent reads calendar events to answer scheduling questions (e.g. "what do I have tomorrow?"). No data is stored or shared.
calendar.events The agent creates calendar events when the user explicitly requests it (e.g. "schedule a meeting with John next Tuesday at 3pm"). Events are only created on user instruction.

API access

The agent exposes a REST API for programmatic use. Authenticate with the x-api-key header.

Send a message

curl -X POST https://your-agent-url/api/chat \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{
    "message": "Summarize the latest emails in my inbox",
    "history": [],
    "gmailToken": "GMAIL_JWT_TOKEN"
  }'

Get a Gmail token for a connected user

curl "https://your-agent-url/api/google-token?email=user@example.com&service=gmail" \
  -H "x-api-key: YOUR_API_KEY"

Response format

{
  "reply": "Here are your latest emails…",
  "toolUses": [
    { "name": "gmail_search", "input": "…", "output": "…" }
  ]
}
